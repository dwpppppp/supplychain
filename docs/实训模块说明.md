本文档分三部分：

# 第一部分 模块说明

## 1.1 模块定位

实训模块把供应链寻源课程中的业务流程做成可交互案例。

- 学生端主入口：`frontend/web/app/(platform)/training/page.tsx`
- 访问路径：`http://localhost:3030/training`
- 已实现案例：草莓果酱采购寻源实训
- 预留案例：电机采购寻源实训、瓶盖采购寻源实训

## 1.2 学生端流程

整个流程线性推进，上一步不完成下一步不解锁。学生选错供应商（不合格）也允许继续走完，只是 Step 02 得 0 分。

**Step 01 需求分析**
学生读一段案例背景，看一张产品配图。例如草莓果酱案例：要采购草莓果酱、年量 120 吨、糖度要 >=65 Brix、不能用人工防腐剂、供应商要有 HACCP 和 ISO22000 认证。点"进入市场寻源"。

**Step 02 市场寻源**
打开一张中国地图，6 家候选供应商以水滴标记散落在不同城市。学生点任一标记，右侧弹出详情面板：图片、所在地、资质认证、产能、报价、风险备注。学生需要靠看认证是否齐全、风险备注是否有红字提示，自己判断是否合格。看中一家后点"选择该供应商，进入谈判"，直接进 Step 03。

**Step 03 商务谈判**
与所选供应商进行 5 轮对话。每轮系统给出 3 个谈判话术选项（如"直接施压要求降价 5%"、"提出年度框架协议换降价"、"询问账期延长"），学生点哪个，供应商就给出对应回复，然后价格/账期/质保对应字段变动。选过的选项置灰不能再选。5 轮结束，最终得到一组成交条款。

**Step 04 采购报价单**
把谈判结果填进一张结构化报价单：主信息（品类、物料、年量、单位、交付周期、有效期、税运方式、供应商、单价、总额、账期、质保）+ 规格明细（草莓案例为糖度、防腐剂、执行标准、认证、验收资料、保质追溯 6 行）。保存到后端。

**Step 05 决策复盘**
展示总分、各步得分拆解、谈判最终条款、报价单关键信息、系统点评。

## 1.3 关键设计取舍

**Step 02 改成选 1 家**
原版是选 3 家入围再选 1 家谈判，多了一层无意义的操作。改成地图上直接选 1 家进入谈判，体验更直接。

**Step 03 所有供应商共享对话脚本**
原版给 A/C/F 三家各写一套话术，工作量 ×3，且学生不会三家都玩，看不到差异。改成共享对话模板，只有初始价格/账期/质保不同，工作量降一个数量级。

**去掉情绪、运费陷阱等复杂规则**
情绪值（mood）、强硬次数累计（hardlineCount）、运费陷阱（freightTrapTriggered）全部移除。这些规则学生玩两次就能找到最优解，教学性弱，维护成本高。

**不新建计分表**
`user_step_submissions` 已有 `score` 字段，建第二张计分表会有数据不一致风险。总分用 `SUM(score) WHERE user_id=X AND case_id=Y` 实时算。

**用 slug 取代硬编码 case_id**
避免删库重建后 id 漂移。前端永远按 `strawberry_jam` 这种业务编码查后端。

# 第二部分 后端概述

## 2.1 表结构总览

9 张实训相关表，分三组：

**第一组：案例配置（教师后台维护）**

- `training_cases` — 案例主表。slug、标题、背景文案、封面图
- `training_suppliers` — Step 02 候选供应商池（每个案例 6 行）。含经纬度、图片、认证、报价文本、风险备注、是否合格
- `training_step_config` — 五步的通用配置。Step 01 文案、Step 04 规格模板（form_schema JSON）

**第二组：Step 03 专用配置**

- `training_case_negotiation_supplier` — 入围供应商的初始值（每个案例 3 行）。初始价格、账期、质保
- `training_case_negotiation_option` — 谈判选项库（每个案例 15 行：5 轮 × 每轮 3 个）。**所有供应商共享同一套对话脚本**，只有初始值不同

**第三组：用户数据**

- `user_training_progress` — 用户在每个案例的当前步骤和状态
- `user_step_submissions` — 每步的提交记录，**同时承担计分职责**
- `training_purchase_quotes` — Step 04 报价单主表
- `training_purchase_quote_specs` — Step 04 报价单规格明细

## 2.2 接口总览

接口前缀统一是 `/api/v1`，所有接口需登录。

**学生端**

| 方法 | 路径 | 作用 |
| --- | --- | --- |
| GET | `/training/cases` | 案例列表 |
| GET | `/training/cases/{case_id}` | 案例详情 |
| GET | `/training/cases/{case_id}/steps/{step}` | 步骤配置 |
| GET | `/training/cases/{case_id}/suppliers` | Step 02 候选供应商（含经纬度） |
| POST | `/training/cases/{case_id}/steps/{step}/submit` | 提交步骤答案 |
| GET | `/training/cases/{case_id}/negotiation/suppliers` | Step 03 谈判初始值 |
| GET | `/training/cases/{case_id}/negotiation/options?round_no={n}` | Step 03 当前轮选项 |
| POST | `/training/cases/{case_id}/negotiation/submit` | Step 03 提交谈判结果 |
| POST | `/training/cases/{case_id}/purchase-quote` | Step 04 保存报价单 |
| GET | `/training/cases/{case_id}/purchase-quote` | 获取当前用户报价单 |
| GET | `/training/cases/{case_id}/report` | Step 05 训练报告（聚合多源数据） |

**管理端**（沿用旧版）

| 方法 | 路径 | 作用 |
| --- | --- | --- |
| GET | `/training/admin/cases` | 管理端案例列表 |
| POST | `/training/admin/cases` | 新建案例配置 |
| GET | `/training/admin/cases/{case_id}` | 获取案例配置 |
| PUT | `/training/admin/cases/{case_id}` | 更新案例配置 |
| POST | `/training/admin/cases/{case_id}/publish` | 发布案例 |
| POST | `/training/admin/cases/{case_id}/unpublish` | 取消发布 |

管理端接口需要用户角色是 `teacher` 或 `admin`。

# 第三部分 开发展开说明

## 3.1 Step 01 需求分析

只保留案例说明和配图，不展示表单或侧边提示。

- 案例文本：`training_cases.background`
- 配图：`training_cases.cover_image_url`，例如 `/static/training/training1.png`
- 前端访问地址：由 `API_BASE_URL` 去掉 `/api/v1` 后拼接

旧版的前端常量 `DEMAND_TEXT` 不再使用，统一从后端读取。

## 3.2 Step 02 市场寻源

### 3.2.1 地图渲染

前端使用 Leaflet + 高德地图瓦片：

- Leaflet 版本：1.9.4
- 瓦片源：`https://webrd0{s}.is.autonavi.com/appmaptile?lang=zh_cn&...`，中文地名，国内访问快
- 标记：自定义 `divIcon`，水滴形状（`border-radius: 50% 50% 50% 0` + `rotate(-45deg)`）
- `iconAnchor` 设为 `[16, 42]`，水滴尖端对准真实坐标

### 3.2.2 视觉规则

- 所有候选供应商在地图上视觉一致（深灰水滴）
- 选中状态：水滴变 IBM Blue 60，外圈高亮
- 不合格供应商不在地图上做区分，仅在详情面板的"风险备注"以红字提示
- 教学目标：让学生通过认证、风险备注自行判断是否合格

### 3.2.3 评分

`user_step_submissions.score` 由后端在提交时计算：

- 选中合格供应商：100 分
- 选中不合格供应商：0 分（仍允许进入下一步）

合格判定来自 `training_suppliers.is_qualified` 字段。

### 3.2.4 草莓案例标准答案

合格的 3 家：A 锐达食品原料、C 优鲜源食品科技、F 盛丰原料（天津）。
不合格的 3 家：B 甜源果业（缺 HACCP/ISO22000）、D 绿野农产（历史质量投诉）、E 康源食品（缺 HACCP）。

## 3.3 Step 03 商务谈判

### 3.3.1 谈判状态

```ts
type NegotiationState = {
  supplierCode: string;
  round: number;          // 1~5
  currentPrice: number;
  paymentDays: number;
  warrantyMonths: number;
  pickedOptionIds: number[];   // 已选过的 option，前端置灰
  done: boolean;
  log: { round: number; ask: string; answer: string }[];
};
```

### 3.3.2 交互流程

1. 从 Step 02 拿到选中的 supplier_code
2. 调 `GET /negotiation/suppliers`，找到对应 supplier_code 的初始值，塞入 state
3. 每轮调 `GET /negotiation/options?round_no=N`，数据库有几条就展示几条
4. 学生点选项 → 显示供应商回复 → 按 `affect_field` 应用 `delta_value` 更新 state
5. 同一 option 选过后前端置灰
6. 5 轮选完后调 `POST /negotiation/submit`，传 picked_option_ids，后端算分并写入 `user_step_submissions`

### 3.3.3 评分

- 价格节省（节省 10% 即满分）：最高 60 分
- 账期改善（每多 1 天 1 分）：最高 20 分
- 质保改善（每多 1 月 2 分）：最高 20 分

具体公式可调，由后端实现，前端不参与计算。

## 3.4 Step 04 采购报价单

### 3.4.1 主表字段

`category` / `item_name` / `annual_quantity` / `unit` / `delivery_cycle` / `quote_validity` / `tax_mode` / `supplier_code` / `supplier_name` / `unit_price` / `estimated_total` / `payment_terms` / `warranty_months` / `remarks`。

### 3.4.2 规格明细结构

```ts
type PurchaseQuoteSpec = {
  orderIndex: number;
  fieldKey: string;
  label: string;
  value: string;
  unit?: string;
  requirementType?: string;
  isRequired: boolean;
};
```

草莓案例规格明细：

| 顺序 | key | 名称 | 默认值 |
| --- | --- | --- | --- |
| 1 | `brix` | 糖度要求 | `>=65 Brix` |
| 2 | `preservative` | 防腐剂要求 | 不得使用人工防腐剂 |
| 3 | `food_standard` | 执行标准 | 符合 GB 2760 |
| 4 | `certification` | 供应商认证 | HACCP 与 ISO22000 均在有效期内 |
| 5 | `inspection_documents` | 验收资料 | 每批提供 COA、批次检测报告和生产日期记录 |
| 6 | `shelf_life` | 保质与追溯 | 到货剩余保质期不少于总保质期的 2/3，支持批次追溯 |

模板存储位置：`training_step_config.form_schema`（step=4），新增案例时换 specs 数组即可，**不改数据库字段**。

### 3.4.3 报价单保存

前端调用 `savePurchaseQuote(caseId, payload, token)`：

- `user_id + case_id` 唯一，再次提交时覆盖主表并重建明细
- 草莓案例 `backendCaseId` 应从 `/training/cases` 接口按 `slug='strawberry_jam'` 拿真实 id，不再硬编码 1

## 3.5 Step 05 决策复盘

复盘展示：

- 总分（各步 score 求和）
- 各步得分拆分
- 谈判最终条款（价格、账期、质保）
- 报价单关键信息
- 系统点评

后端 `GET /training/cases/{case_id}/report` 聚合：

- `user_step_submissions` 各步 `score` / `dimension_scores` / `result_snapshot`
- `training_purchase_quotes` + `_specs`

## 3.6 表字段细节

### 3.6.1 删除并重建顺序

所有实训相关表全部删除重建。删除顺序按依赖反向：

```
training_purchase_quote_specs
training_purchase_quotes
user_step_submissions
user_training_progress
training_case_negotiation_option
training_case_negotiation_supplier
training_step_config
training_suppliers

```

### 3.6.2 关键字段

**`training_cases` 新增字段**

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `slug` | `VARCHAR(64) UNIQUE` | 业务编码，如 `strawberry_jam` |
| `cover_image_url` | `VARCHAR(512)` | Step 01 封面图 |

**`training_suppliers` 新增字段**

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `latitude` | `DECIMAL(10,7)` | 纬度，地图标点用 |
| `longitude` | `DECIMAL(10,7)` | 经度 |
| `image_url` | `VARCHAR(512)` | 供应商图片/Logo |
| `is_qualified` | `TINYINT(1)` | 是否合格（影响 Step 02 评分） |
| `price_text` | `VARCHAR(64)` | 报价文本 |
| `risk_note` | `VARCHAR(512)` | 风险备注，详情面板展示 |

**`training_case_negotiation_supplier`（新表）**

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `case_id` | `BIGINT` | |
| `supplier_code` | `VARCHAR(16)` | 与 `training_suppliers.code` 对齐 |
| `supplier_name` | `VARCHAR(64)` | |
| `init_price` | `DECIMAL(12,2)` | 初始报价 |
| `init_payment_days` | `INT` | 初始账期 |
| `init_warranty_months` | `INT` | 初始质保 |

**`training_case_negotiation_option`（新表，案例级共享，不绑定 supplier_code）**

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `case_id` | `BIGINT` | |
| `round_no` | `SMALLINT` | 第几轮（1-5） |
| `ask` | `TEXT` | 学生话术 |
| `answer` | `TEXT` | 供应商回复 |
| `affect_field` | `VARCHAR(32)` | `price` / `payment_days` / `warranty_months` |
| `delta_value` | `DECIMAL(12,2)` | 变动值，负数=降价 |
| `sort_order` | `INT` | 同轮内展示顺序 |

**`user_step_submissions` 承担计分职责的字段**

| 字段 | 用途 |
| --- | --- |
| `score` | 本步得分（`DECIMAL(6,2)`） |
| `dimension_scores` | JSON 拆分，如 `{"price":45,"payment":18,"warranty":16}` |
| `result_snapshot` | 最终状态快照，如谈判结束时的 price / payment_days |

总分通过 `SUM(score) WHERE user_id=X AND case_id=Y` 计算，不冗余存计分表。

## 3.8 开发坑点

1. **不要把规格字段写死到后端列里**
   主表只存通用字段，具体规格走 `training_purchase_quote_specs`。

2. **不同案例规格不同**
   通过 `order_index + field_key + label + value` 表达，新增案例时只换数据不改字段。

3. **案例 ID 不再硬编码**
   前端按 `slug` 查询，`training_cases.slug` 已加 UNIQUE 约束。

4. **谈判选项是案例级共享**
   `training_case_negotiation_option` 不绑定 `supplier_code`，所有入围供应商共用一套对话模板。换供应商只换初始值。

5. **不新建计分表**
   `user_step_submissions.score` + `dimension_scores` 已经够用，总分用 SUM 实时算。

6. **地图组件必须客户端渲染**
   Next.js 中用 `dynamic(() => import('...'), { ssr: false })`，否则 SSR 会因 `window` 未定义报错。

7. **后端通过 `init_db()` 自动 `create_all`**
   重建后首次启动会按新 models 建表。重建前务必备份现有数据或确认可清空。

8. **前端登录失效自动清理顶部状态**
   API 请求遇到 401 触发全局会话过期事件，右上角回到未登录状态。

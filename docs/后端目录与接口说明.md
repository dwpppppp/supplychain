## 3. 分层职责说明

### 3.1 `main.py`

负责：

- 创建 FastAPI 应用
- 初始化数据库
- 配置 CORS
- 注入 Bearer Token 的 OpenAPI 安全定义
- 注册所有 `/api/v1/*` 路由
- 统一异常处理

### 3.2 `app/core`

基础设施层，主要承接“全项目通用能力”：

- `config.py`：读取应用名、版本号、数据库、Redis、DashScope、OSS 等配置
- `database.py`：创建异步数据库连接和 `AsyncSession`
- `dependencies.py`：提供 `get_current_user`、`get_optional_user`
- `middleware.py`：聊天限流、消息去重、TTS/Redis 缓存
- `response.py`：统一返回 `{ code, message, data }`
- `security.py`：JWT 与密码处理

### 3.3 `app/models`

ORM 模型层，定义用户、课程章节、知识图谱、对话、消息、实训案例、埋点事件等数据库实体。

### 3.4 `app/schemas`

请求和响应的数据结构定义层。路由入参、业务出参、前后端约定字段都优先在这里统一。

### 3.5 `app/crud`

只处理“数据库读写”，不放复杂业务判断。适合承接：

- 查询列表
- 按条件检索
- 插入/更新实体
- 统计聚合

### 3.6 `app/engine`

外部能力封装层，主要给 AI 相关业务复用：

- `llm.py`：文本生成、流式输出
- `rag.py`：从课程/图谱上下文中检索参考知识
- `tts.py`：把数字人回复合成为音频
- `oss_client.py`：上传音频、生成可访问链接

### 3.7 `app/services`

业务编排层。路由调用 service，service 再调 crud / engine / schema / model：

- `auth_service.py`：注册、登录、刷新 token、获取当前用户
- `ai_service.py`：聊天会话、消息发送、流式回复、数字人音频、ASR/TTS 相关
- `course_service.py`：课程目录、章节详情、学习笔记、阅读进度、知识图谱
- `training_service.py`：实训案例、步骤、谈判模拟、报告

### 3.8 `app/routers`

最终暴露给前端/客户端的 HTTP 接口层。每个模块一个 router，统一在 `main.py` 中挂到 `/api/v1`。

| 方法 | 路径 | 是否登录 | 作用 | 主要文件 |
| --- | --- | --- | --- | --- |
| POST | `/chat/conversations` | 是 | 创建会话 | `routers/chat_router.py` → `services/ai_service.py` |
| GET | `/chat/conversations` | 是 | 获取会话列表 | `routers/chat_router.py` → `services/ai_service.py` |

| POST | `/chat/messages` | 是 | 发送普通消息，返回完整结果；数字人模式会返回文本 + 音频链接 | `routers/chat_router.py` → `services/ai_service.py` → `engine/llm.py` / `engine/tts.py` / `engine/oss_client.py` |

| POST | `/chat/asr` | 是 | 语音转文字 | `routers/chat_router.py` |

#### 关键请求结构

- `ConversationCreate`
  - `scene`：默认 `assistant`
- `SendMessageRequest`
  - `conversation_id`
  - `content`
  - `response_mode`
    - `text`：普通文本问答
    - `voice`：数字人语音问答
- `ASRRequest`
  - `audio_data`：base64
  - `format`
  - `sample_rate`

#### 关键说明

- `/chat/messages` 适合数字人和非流式回复
- `/chat/messages/stream` 适合主聊天窗口流式打字效果
- 聊天接口进入前会走 `check_chat_rate_limit`
- 消息、TTS、去重缓存相关逻辑集中在 `app/core/middleware.py`

## 6. 当前后端接口组织建议

为了让后续维护更清晰，建议继续保持下面这条链路：

```text
router -> service -> crud / engine -> model / schema
```

具体原则：

- `router` 只做参数接收、鉴权依赖、返回格式包装
- `service` 负责业务流程和模块编排
- `crud` 只做数据库读写
- `engine` 只做 AI、TTS、OSS、RAG 这类外部能力封装
- `schema` 统一前后端数据契约

这样前端以后查接口时，可以直接按“路由文件 -> service 文件 -> schema 文件”的顺序定位。

### 8.2 `/course/knowledge-graph` 响应字段约定

后端统一返回 `{ code, message, data }`，其中 `data` 对应：

```ts
type KnowledgeGraphData = {
  nodes: {
    id: string;
    title: string;
    description?: string;
    icon?: string;
    node_type: string;
    chapter_id?: number;
  }[];
  edges: {
    source: string;
    target: string;
    relation_type: string;
  }[];
};
```

前端当前依赖的 `node_type`：

- `chapter`：章节根节点
- `concept`：核心概念
- `method`：方法
- `step`：流程步骤
- `tool`：工具
- `metric`：指标

前端当前依赖的 `relation_type`：

- `contains`
- `prerequisite`
- `related`
- `applied_in`

### 8.3 后端维护注意事项

- `KnowledgeNodeOut.id` 必须稳定，前端使用它作为 SVG 节点 key、边 source/target 匹配依据。
- `KnowledgeEdgeOut.source` / `target` 必须能在 nodes 中找到对应 id，否则前端会跳过该边。
- `title` 当前会完整展示在节点下方，不再由前端截断；过长标题建议后端保持业务可读但避免冗余描述。
- `description` 会展示在右侧节点详情面板中，可以为空。
- 图谱接口目前允许游客访问；章节详情、笔记、进度等仍按原权限控制。
- 当前图谱初始化 SQL 仍主要在 `backend/sql/update_knowledge_graph_chapter1.sql` 等脚本中维护，新增课程内容时应同步补齐节点和关系。
